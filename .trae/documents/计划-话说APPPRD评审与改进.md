## 2. 当前状态分析（基于仓库现状）

### 仓库现状
- 当前目录仅包含 1 份 PRD 文件：[话说APP需求文档（PRD）-0407.md](file:///e:/learning/web/260406/whisper/%E8%AF%9D%E8%AF%B4APP%E9%9C%80%E6%B1%82%E6%96%87%E6%A1%A3%EF%BC%88PRD%EF%BC%89-0407.md)。
- 未发现工程代码、依赖清单、数据库 schema、API 定义或 UI 设计稿，因此本次评审聚焦 PRD 的可实现性与裁剪可行性，而非代码层面审查。

### 用户已确认的关键约束（用于锁定方案）
- 上线范围：iOS + Android 双端（Flutter 合理）。
- V1.0 数据形态：纯本地单机（不依赖云同步/账号体系）。
- AI 依赖：先不接 AI（可以用模板/规则替代）。
- 时间预算：不设死期（以每周可交付迭代为主）。

### PRD 的优势（可行性加分项）
- “端重云轻”“降级策略”“反焦虑规范”写得清晰，适合新人把控复杂度与体验一致性。
- 以“本地身份隔离”为核心差异点，具备可验证的产品壁垒（尤其在单机阶段）。

### PRD 的主要风险点（新人独立开发视角）
- **范围偏大**：PRD 虽标注 MVP，但仍包含云同步、异步共鸣池、内容脱敏、内容安全护栏、AI 卡片等多链路；单人同时做体验+数据+合规容易失控。
- **单机与“社区共鸣”的目标存在张力**：纯本地单机无法产生真实的“同频共鸣池”，需要在 V1.0 明确定位为“匿名身份隔离的情绪记录/卡片生成工具”，或接受 V1.1 再引入联网共鸣。
- **合规与内容安全的隐性成本**：一旦引入 UGC 上云与用户互动，就会引入审核、举报、风控、留痕/协查接口等工作量；这对新人独立开发是最大的不可控项之一。

## 3. 拟定改进方向（决策已闭合，执行不再需要额外选择）

### 3.1 重新定义 V1.0：单机“情绪标本”闭环（建议）
将 V1.0 从“匿名社区”调整为“匿名身份隔离 + 情绪记录 + 卡片生成/分享”的单机闭环：
- 目标：验证“无压力表达 + 仪式感卡片”的核心价值，先做可持续交付的产品壳与体验规范。
- 不做：任何需要服务器的内容流/互动/同步/推送/审核后台。

### 3.2 P0 功能裁剪（V1.0 仅保留 2 条核心链路）
链路 A：身份隔离 → 发布文字心事 → 本地列表/详情 → 归档/销毁  
链路 B：发布事件 → 生成情绪卡片（模板/规则）→ 本地保存 → 导出图片分享

对应模块边界（写入 PRD 的“明确不做清单”）：
- 不做：Supabase 同步、异步共鸣池、回声/书签/点亮、举报与人工审核流程、语音发布、AI 生成文案（全部延期到 V1.1+）。
- 可选但建议延期：敏感词脱敏（若只本地保存且不联网，可降为“手动提示”）。

### 3.3 数据与存储（V1.0 单机实现口径）
- SQLite（或同等本地 DB）按 identity_hash 物理分区/逻辑隔离。
- 事件与队列：V1.0 不需要“后台静默队列”，改为“本地事件触发器 + 失败重试一次 + 永不阻塞主线程”。
- 加密优先级：身份 vault 与私密归档优先；图片导出默认去 EXIF（若实现成本可控）。

### 3.4 体验规范落地（把 PRD 的“规范”变成“可验收条目”）
- 禁推送/禁红点/禁强提醒：在验收标准里明确“应用内唯一提示方式”（如 toast/snackbar）。
- 动效：统一时长与曲线；优先实现“发布后轻反馈 + 卡片生成过渡”，其余延后。
- 性能口径：用可测量指标替换主观描述（如：冷启动、首屏、列表滚动帧率、导出图片耗时）。

## 4. 计划内的具体工作（评审+落地改进）

### 4.1 PRD 评审输出（不改代码也能完成）
- 给出一份“可行/不可行”结论：以单机 V1.0 为前提，哪些条款可直接实现，哪些需要改写或延期。
- 形成“范围裁剪表”（P0/P1/P2）与“明确不做清单”（避免中途失控）。
- 为每个 P0 模块补齐：用户故事、主流程、异常流程、数据字段、验收标准。

### 4.2 对 PRD 的具体改动建议（执行阶段将直接改 PRD 文件）
对 [话说APP需求文档（PRD）-0407.md](file:///e:/learning/web/260406/whisper/%E8%AF%9D%E8%AF%B4APP%E9%9C%80%E6%B1%82%E6%96%87%E6%A1%A3%EF%BC%88PRD%EF%BC%89-0407.md) 的改动范围（按章节）：
- 架构章节：将“端重云轻”改为“纯本地单机”，把 Supabase/OpenRouter 放入 V1.1+ 选项。
- 功能章节：将 3.2（异步共鸣池）、3.3（AI 卡片）中依赖云/AI 的描述改为模板/规则实现，并标注延期项。
- 非功能性需求：把“合规/审核/监管协查接口”改为“V1.0 不产生线上 UGC，因此不涉及审核后台；V1.1 引入 UGC 时再补齐合规模块”。
- 指标章节：将 DAU/留存类指标保留为参考，但新增单机可统计指标（如：发布次数、卡片导出率、身份切换使用率、归档/销毁使用率）。

## 5. 假设与决策（已锁定）
- 假设：V1.0 允许“无服务器”发布，因此不存在“社区内容流”，以本地记录为主。
- 决策：AI 相关能力全部用本地模板/规则替代；将“AI 优雅降级”调整为“模板优雅降级（默认即模板）”。
- 决策：双端上线不影响单机架构，但验收需覆盖 iOS/Android 的图片导出与权限差异。

## 6. 验证方式（评审与执行阶段都可用）
- PRD 一致性检查：每个 P0 功能在文档中都能找到入口、流程、数据、验收标准四件套。
- 可实现性检查：每个 P0 功能都能映射到“Flutter UI + 本地 DB + 本地文件导出”三类能力，不引入服务器依赖。
- 风险检查：任何涉及 UGC 上云、审核、推送、实时通信的内容都必须明确标注为 V1.1+。
