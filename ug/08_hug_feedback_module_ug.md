# 拥抱回流模块使用说明

## 1. 模块介绍

拥抱回流模块（Hug Feedback Module）是话说APP的社交反馈模块，提供分享卡片的拥抱计数和温暖反馈功能，形成"记录→分享→拥抱→回流"的完整闭环。

### 核心特性

- **拥抱计数**：统计分享卡片收到的拥抱数量
- **批量拉取**：批量获取已分享卡片的最新拥抱数据
- **温暖反馈**：显示"收到X个拥抱"的通知提示
- **防刷机制**：每日每设备仅能拥抱同一张卡片1次
- **冷启动触发**：应用启动时自动拉取最新拥抱数据
- **进入回声页触发**：进入回声页面时拉取最新数据

## 2. 使用方法

### 2.1 进入回声页面

在底部导航栏中点击"💫 回声"标签，即可进入回声页面。进入时系统会自动拉取你所有已分享卡片的最新拥抱数据。

### 2.2 查看拥抱反馈

在回声页面中，你可以看到：

- **拥抱通知**：页面顶部显示温暖的拥抱通知
  - 🤗 "收到了x个陌生人的拥抱"
  - 🌟 "有人正在想你"
  - 💝 "你的心事被人看到了"
  - ☀️ "温暖正在路上"
- **已分享卡片列表**：显示所有分享过的卡片及其收到的拥抱数量

### 2.3 卡片详情查看

点击已分享的卡片，可以查看：
- AI生成的诗句
- 收到的拥抱数量
- 分享时间

### 2.4 外部用户如何送拥抱

当你的卡片被分享到微信/小红书等平台后：

1. 熟人看到分享的卡片图片
2. 扫描卡片上的二维码
3. 打开分享落地页，看到AI诗句
4. 点击"🫂 送拥抱"按钮
5. 完成拥抱，计数+1

> 注意：每个设备每天只能对同一张卡片送1次拥抱

## 3. 工作原理

### 3.1 数据流

```
用户分享卡片 → 创建分享记录(Supabase) → 生成二维码
      ↓
熟人扫码 → 打开落地页 → 点击送拥抱 → 更新拥抱计数
      ↓
用户打开回声页 → 批量拉取拥抱数据 → 显示温暖通知
```

### 3.2 自动触发时机

拥抱数据会在以下时机自动拉取：

| 触发时机 | 说明 |
|---------|------|
| 应用启动（冷启动） | 首次打开应用时自动拉取 |
| 进入回声页面 | 每次进入回声页时拉取最新数据 |
| 分享成功后 | 分享卡片后自动刷新 |

### 3.3 本地缓存机制

- **缓存有效期**：30分钟
- **缓存内容**：已分享卡片的share_id列表
- **缓存位置**：localStorage `huashuo_shared_cards`
- **自动清理**：超过有效期后自动重新拉取

## 4. 常见问题

### Q: 为什么回声页面没有显示拥抱数据？

A: 请确认以下条件：
1. 你已经成功分享过卡片（在卡片页点击分享按钮）
2. 已经有熟人通过扫描二维码送过拥抱
3. 网络连接正常

### Q: 拥抱通知显示的数字不准确？

A: 拥抱数据有30分钟的缓存有效期。如果数据不准确，可以：
1. 下拉刷新回声页面
2. 等待30分钟后缓存自动更新
3. 重新进入回声页面触发拉取

### Q: 为什么同一张卡片我只能送一次拥抱？

A: 为了防止刷拥抱，系统设置了防刷机制：每个设备每天只能对同一张卡片送1次拥抱。这是通过设备ID来识别的。

### Q: 分享的卡片二维码扫描后打不开？

A: 请确认：
1. 卡片已经成功创建分享记录（分享时会调用后端API）
2. 网络连接正常
3. 二维码没有损坏或遮挡

### Q: 我没有分享过卡片，但回声页面显示有拥抱？

A: 这种情况不会发生。回声页面只显示你主动分享过的卡片的拥抱数据。

### Q: 如何查看是谁送的拥抱？

A: 出于隐私保护，拥抱是匿名的。你只能看到拥抱的数量，无法看到具体是谁送的。

## 5. 业务规则

### 5.1 拥抱防刷规则

- 每个设备每天对同一张卡片只能送1次拥抱
- 设备ID通过localStorage持久化，保持稳定
- 防刷记录存储在Supabase的hugs表中

### 5.2 数据拉取规则

- 批量拉取上限：每次最多100个share_id
- 拉取频率：每次进入回声页面或应用冷启动时
- 缓存有效期：30分钟

### 5.3 拥抱计数规则

- 拥抱计数存储在Supabase的shared_cards表中
- 每次成功拥抱后计数+1
- 计数不可减少（不支持取消拥抱）

## 7. 错误处理

### 7.1 常见错误及处理

| 错误场景 | 处理方式 |
|---------|---------|
| 网络请求失败 | 使用本地缓存数据，显示离线提示 |
| Supabase不可用 | 返回模拟数据（hugs_count: 0） |
| 分享卡片不存在 | 返回404错误，提示"分享卡片不存在" |
| 拥抱请求过于频繁 | 返回429错误，提示"请求过于频繁" |
| 批量查询ID过多 | 返回400错误，提示"列表长度不能超过100" |

### 7.2 降级策略

当后端服务不可用时：
- 拥抱计数显示为0
- 拥抱操作返回模拟成功（hugs_count: 1）
- 批量查询返回全0数据

## 10. 注意事项

1. **网络依赖**：拥抱数据存储在云端（Supabase），需要网络连接才能获取最新数据
2. **隐私保护**：拥抱是匿名的，不会显示送拥抱者的身份信息
3. **防刷机制**：同一设备每天只能对同一张卡片送1次拥抱，请珍惜每次拥抱
4. **缓存更新**：拥抱数据有30分钟缓存，如需立即查看最新数据请手动刷新
5. **离线使用**：在离线状态下，回声页面会显示上次缓存的数据
6. **数据同步**：多设备间的数据通过Supabase云端同步，但本地缓存不跨设备共享

## 11. 总结

拥抱回流模块是话说APP"记录→分享→拥抱"闭环的关键环节。通过拥抱计数和温暖通知，让用户感受到来自陌生人的关心和支持，增强应用的情感连接价值。

该模块支持批量数据拉取、冷启动触发、防刷保护等特性，在保证数据准确性的同时提供了良好的用户体验。
