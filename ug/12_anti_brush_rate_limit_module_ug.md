# 防刷限流模块使用说明

## 1. 模块介绍

防刷限流模块（Anti-brush Rate Limit Module）是话说APP的安全保障模块，用于防止恶意刷取API和重复点击拥抱按钮，保护系统安全和资源。

### 核心特性

- **IP限流**：限制同一IP的请求频率
- **设备限流**：限制同一设备的请求频率
- **路径限流**：针对不同API路径设置不同的限流规则
- **拥抱防刷**：防止同一设备对同一分享卡片重复点击拥抱
- **异常检测**：检测并阻止异常请求模式
- **过期数据清理**：自动清理过期的限流数据
- **统计信息**：提供限流统计信息

## 3. 核心功能使用

### 3.1 基础限流使用

```python
from app.utils.rate_limiter import RateLimiter

# 创建限流实例
rate_limiter = RateLimiter()

# 检查IP限流
ip = "192.168.1.1"
if not rate_limiter.is_ip_allowed(ip):
    print("IP请求过于频繁")

# 检查设备限流
device_id = "test-device-123"
if not rate_limiter.is_device_allowed(device_id):
    print("设备请求过于频繁")

# 检查路径限流
path = "/api/ai/translate"
limits = {
    "/api/ai/translate": 5,
    "/api/share/create": 10,
    "/api/share/hug": 3,
    "/api/share/batch": 5
}
if not rate_limiter.is_path_allowed(ip, path, limits):
    print("路径请求过于频繁")

# 检查拥抱防刷
share_id = "test-share-123"
if not rate_limiter.is_device_allowed(device_id, share_id):
    print("您今天已经给这个卡片送过拥抱了")

# 清理过期数据
rate_limiter.clear_expired()

# 获取统计信息
stats = rate_limiter.get_stats()
print(stats)
```

## 4. 配置选项

### 4.1 限流规则

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| IP限流 | 3次/分钟 | 同一IP对同一API的请求频率限制 |
| 设备限流 | 1次/天 | 同一设备对同一分享卡片的拥抱频率限制 |
| 路径限流 | 见下表 | 不同API路径的请求频率限制 |

### 4.2 路径限流规则

| API路径 | 限制次数/分钟 | 说明 |
|---------|--------------|------|
| `/api/ai/translate` | 5 | AI翻译接口 |
| `/api/share/create` | 10 | 创建分享接口 |
| `/api/share/hug` | 3 | 拥抱接口 |
| `/api/share/batch` | 5 | 批量获取接口 |
| 其他路径 | 3 | 默认限制 |

### 4.3 时间窗口

| 限流类型 | 时间窗口 | 说明 |
|---------|---------|------|
| IP限流 | 60秒 | 1分钟窗口 |
| 设备限流 | 86400秒 | 24小时窗口 |
| 路径限流 | 60秒 | 1分钟窗口 |
| 拥抱防刷 | 86400秒 | 24小时窗口 |

## 5. 业务规则

### 5.1 限流规则

- **IP限流**：同IP对同一API每分钟最多3次
- **设备限流**：同设备对同一分享卡片每日最多1次拥抱
- **路径限流**：根据不同API路径设置不同的限制
- **响应处理**：超过限制时返回429状态码

### 5.2 错误处理

- **限流触发**：返回429状态码，提示"请求过于频繁，请稍后再试"
- **拥抱防刷**：抛出异常，提示"您今天已经给这个卡片送过拥抱了"

### 5.3 数据清理

- 自动清理过期的限流数据，避免内存占用过高
- 每次请求时都会清理过期数据

## 6. 性能优化

### 6.1 内存管理

- 使用过期数据清理机制，避免内存溢出
- 生产环境建议使用Redis作为缓存

### 6.2 性能考虑

- 限流检查响应时间 < 10ms
- 并发处理能力 ≥ 1000 QPS
- 适合高并发场景

### 6.3 优化建议

- **生产环境**：使用Redis作为缓存，提高性能和可靠性
- **定期清理**：设置定时任务定期清理过期数据
- **监控**：监控限流触发情况，及时调整限流规则

## 10. 常见问题与解决方案

### 10.1 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 请求被限流 | 请求频率过高 | 降低请求频率，等待限流窗口过期 |
| 拥抱被拒绝 | 24小时内已拥抱过 | 等待24小时后再拥抱 |
| 内存占用高 | 过期数据未清理 | 生产环境使用Redis，定期清理数据 |
| 限流误判 | 限流阈值设置不合理 | 调整限流阈值，根据实际业务情况优化 |

### 10.2 解决方案

1. **请求被限流**：
   - 降低请求频率
   - 等待限流窗口过期
   - 检查是否有异常请求导致限流

2. **拥抱被拒绝**：
   - 等待24小时后再拥抱
   - 检查设备ID是否正确
   - 确认是否真的需要重复拥抱

3. **内存占用高**：
   - 生产环境使用Redis作为缓存
   - 定期清理过期数据
   - 优化限流规则，减少数据存储

4. **限流误判**：
   - 调整限流阈值，根据实际业务情况优化
   - 对特殊场景设置白名单
   - 监控限流触发情况，及时调整

## 11. 生产环境建议

### 11.1 部署建议

- **使用Redis**：生产环境建议使用Redis作为缓存，提高性能和可靠性
- **集群部署**：多实例部署时，使用Redis共享限流数据
- **监控告警**：设置限流触发告警，及时发现异常

### 11.2 配置建议

- **合理阈值**：根据实际业务情况调整限流阈值
- **定期清理**：设置定时任务定期清理过期数据
- **日志记录**：记录限流触发情况，用于分析和优化

### 11.3 安全建议

- **防止绕过**：使用设备ID和IP双重限制，防止绕过
- **防刷机制**：针对敏感操作（如拥抱）设置更严格的限制
- **监控异常**：监控异常请求模式，及时发现恶意行为
